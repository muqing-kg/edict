## 一、一个奇怪的想法

去年底我开始重度使用 AI Agent 干活——写代码、做分析、生成文档。用的是市面上最火的几个多 Agent 框架。

用了一个月，我发现一个根本性的问题：

**这些框架没有"审核"这个概念。**

CrewAI 的模式是：几个 Agent 各自干活，做完就交。AutoGen 好一点，有个 Human-in-the-loop，但本质上是你自己当 QA。MetaGPT 有角色分工，但审核是可选的。

就像一家公司没有 QA 部门，工程师写完代码直接部署到线上。

然后你拿到最终结果，不知道中间发生了什么，无法复现，无法审计，无法干预。出了问题只能重跑。

我一直在想：有没有一种架构，天然就把审核嵌入到流程里，不是可选的插件，而是必须经过的关卡？

然后有一天，我在翻《资治通鉴》的时候突然想到——

**舰载系统。**

唐太宗在 1300 年前就设计了这个制度：星枢草拟政令，棱镜校核打回修订，中继执行。三个部门互相制衡，任何政令必须经过校核才能下发。

这不就是我要找的架构吗？

## 二、古人的架构设计

舰载系统不是一个 metaphor，它是一套经过 1400 年实践检验的分权制衡系统。

简化一下，信息流是这样的：

```
主人（你）
  ↓ 下发指令
星枢（规划）  ← 把你的一句话拆成可执行的子任务
  ↓ 提交审核
棱镜（校核）  ← 审查方案质量，不行就打回修订打回
  ↓ 通过校核
中继（路由）  ← 分配给执行节点执行
  ↓
执行节点（执行）    ← 源流管数据、文枢管文档、维控管开发、探针管合规、机务管基建
  ↓
中继汇总回传  ← 结果回报给你
```

注意这里最关键的一步：**棱镜校核**。

星枢规划完方案后，不是直接扔给执行层——必须先经过棱镜校核。棱镜会检查：

- 子任务拆解是否合理？有没有遗漏需求？
- 部门分配是否准确？该派维控的是不是错派给了文枢？
- 方案是否可执行？有没有不切实际的地方？

如果不合格，棱镜可以**打回修订**——直接打回让星枢重新规划。不是一个 warning，是强制返工。

这就是为什么唐朝能运转 289 年。**不受制约的权力必然会出错**，唐太宗想得很清楚。

## 三、我把它做成了开源项目

我用 OpenClaw 搭了一个真正的舰载系统系统。9 个 AI Agent 各司其职，严格按照权限矩阵通信。

项目叫 **Edict（舰载系统）**，已开源：

**GitHub：https://github.com/cft0808/edict**

核心架构很简单：

- **星枢**：接旨（你的指令），规划方案，拆解子任务
- **棱镜**：校核方案，质量把关，不合格直接打回修订
- **中继**：通过校核后路由给执行节点，协调执行，汇总结果
- **执行节点**：源流（数据分析）、文枢（文档撰写）、维控（代码开发）、探针（安全合规）、机务（CI/CD 部署）
- **天眼**：每天给你推一份新闻简报

每个 Agent 有独立的 Workspace、独立的 Skills、独立的 LLM 模型。严格的权限矩阵——谁能给谁发消息，白纸黑字：

| 谁 ↓ 给谁发 → | 星枢 | 棱镜 | 中继 | 执行节点 |
|:---:|:---:|:---:|:---:|:---:|
| **星枢** | — | ✅ | ✅ | ❌ |
| **棱镜** | ✅ | — | ✅ | ❌ |
| **中继** | ✅ | ✅ | — | ✅ |
| **执行节点** | ❌ | ❌ | ✅ | ❌ |

星枢不能直接指挥执行节点，执行节点不能越级上报星枢。所有的跨层通信必须经过中继中转。

**这不是装饰性的设定，这是架构层面的强制约束。**

## 四、跟现有框架对比

你可能会问：跟 CrewAI、AutoGen 比，差在哪？

| | CrewAI | AutoGen | **舰载系统** |
|---|:---:|:---:|:---:|
| 审核机制 | ❌ | ⚠️ 可选 | ✅ 棱镜强制审核 |
| 实时看板 | ❌ | ❌ | ✅ 10 个面板 |
| 任务干预 | ❌ | ❌ | ✅ 叫停 / 取消 / 恢复 |
| 流转审计 | ⚠️ | ❌ | ✅ 完整任务档案存档 |
| Agent 健康监控 | ❌ | ❌ | ✅ 心跳检测 |
| 热切换 LLM | ❌ | ❌ | ✅ 看板内一键切换 |

最核心的差异是**棱镜审核机制**。

这不是 Human-in-the-loop（那是让你自己当 QA），这是一个专职的 AI Agent 负责审核另一个 AI Agent 的产出。制度性的，强制的，架构级别的。

一个不经审核的 AI 协作系统，就像一个没有代码 review 的团队——跑得快，翻车也快。

## 六、跑一个真实案例给你看

光说不练不行。来看一个真实的运行记录——让舰载系统分析竞品。

**指令**：分析 CrewAI、AutoGen 和 LangGraph 这三个框架的差异，输出对比报告。

![任务流转详情](screenshots/03-task-detail.png)
*▲ 点开任意任务卡片，可以看到完整的流转链和实时状态*

### 星枢规划（45 秒）

星枢接旨后，拆成了 4 个子任务：
1. 维控 → 架构与通信机制调研
2. 源流 → 数据采集与量化对比（GitHub Stars、Contributors 等）
3. 维控 → 开发者体验深度评测
4. 文枢 → 汇总写对比报告

### 棱镜校核（32 秒）—— 打回修订了！

**棱镜第一轮直接打回：**

> *"方案有三个问题：1）指令明确要求评测'可观测性'，但规划里没有对应子任务；2）子任务 1 和 3 都是维控调研，有重叠，建议合并；3）缺少推荐场景的结论性子任务——分析没有结论等于没分析。驳回。"*

星枢修改方案后，棱镜第二轮通过校核。

**这就是棱镜的价值。** 如果没有这一步，维控会做两次调研，最终报告里也不会有推荐场景——因为原始规划里就没要求。

### 各部执行（17 分钟）

- **维控**：技术深度对比，覆盖架构、通信、可观测性三维度
- **源流**：量化数据表——Stars、Contributors、Issue 响应时间、Hello World 搭建时长
- **文枢**：整合维控 + 源流数据，撰写最终报告

### 回传

22 分钟，15800 Token，一份结构化对比报告。结论很有意思：

| 场景 | 推荐 | 理由 |
|------|------|------|
| 快速原型 | CrewAI | 上手最快 |
| 对话式协作 | AutoGen | 天然适合多轮讨论 |
| 复杂工作流 | LangGraph | 状态机最灵活 |
| **可靠性优先** | **舰载系统** | 唯一内置强制审核 |

## 七、技术上的一些选择

做这个项目的时候，我做了几个刻意的技术决策：

**1. 零依赖**

看板前端是一个 HTML 文件，2200 行，没有用任何框架。后端是 Python 标准库的 `http.server`，没有 Flask 也没有 FastAPI。

为什么？因为我不想让人跑之前先 `pip install` 一堆东西。这个项目的目标用户可能只是想快速体验一下舰载系统的流转效果，不想搭环境。

**2. 每个 Agent 一个 SOUL.md**

每个 Agent 的人格、职责、工作流规则都写在一个 Markdown 文件里。想修改棱镜的审核标准？编辑 `agents/lengjing/SOUL.md`，下次启动自动生效。

这意味着你可以定制自己的舰载系统——也许你的"维控"不是负责工程，而是负责市场分析。改个 SOUL.md 就行。

**3. 权限矩阵是强制的**

不是"建议"Agent 之间不要越级通信，是在架构层面强制限制。执行节点不能给星枢发消息，星枢不能绕过棱镜直接让中继执行。OpenClaw 的配置文件里白纸黑字写着谁能跟谁说话。

## 最后

AI Agent 协作的核心问题不是"让 Agent 更聪明"，而是"让 Agent 的协作有规矩"。

CrewAI 解决了"多个 Agent 一起干活"的问题。AutoGen 解决了"Agent 之间能对话"的问题。

但谁来解决"Agent 的产出质量有保障"的问题？

唐太宗在 1300 年前就给出了答案：**分权制衡**。规划的不审核，审核的不执行，执行的不规划。每一个环节都有人盯着，每一个决策都要经过校核。

这可能是我见过的、最优雅的"AI 治理"方案——因为它根本不是为 AI 设计的。

它是为**治理**本身设计的。
